# 🟢 pytest란? (초딩용)

> "코드가 잘 돌아가는지 자동으로 검사해주는 도구. 한 줄로 짤 수 있어요."

생성: 2026-05-11 · 출처: `/jooladen3-good-question pytest`

---

## 1. 왜 나왔어? — "옛날 옛적에 개발자들이 너무 힘들었어요"

옛날(2000년대 초반) 파이썬으로 코드 검사(테스트)를 짜려면 정말 귀찮았어요. **테스트 한 개 만들려고 의식 절차를 5단계나 거쳐야** 했거든요.

**비유**: 지금 친구한테 "이거 맞나?" 한 마디 물어보는 게 자연스럽잖아요? 그런데 옛날 방식은 **"안녕하십니까. 본인은 ○○동에 거주하는 △△△이며, 다음 사항에 대해 정중히 질문드리는 바입니다..."** 이렇게 시작해야 했어요. 매번.

가장 답답했던 일:
- 리스트 두 개를 비교했는데 **"틀렸다"** 만 알려주고 어디가 다른지는 안 알려줌. 직접 눈으로 비교해야 했어요.
- 같은 검사를 100가지 데이터로 돌리고 싶으면? **메서드를 100번 복사 붙여넣기** 해야 했어요.
- 검사 짜는 게 너무 귀찮으니 → **결국 안 짜게 됨** → 새벽 3시에 코드가 폭발 💥

## 2. 이게 뭐야? — "포스트잇 한 장이에요"

pytest는 **파이썬으로 짠 코드가 잘 돌아가는지 자동으로 확인해주는 도구**예요.

**비유**: 옛날 도구(unittest)가 **관공서에 민원 신청할 때 쓰는 빳빳한 양식 종이**(접수번호·도장·서명·날짜 다 적어야 함)라면, pytest는 **포스트잇에 한 줄 써서 모니터에 딱 붙이는 메모**예요.

같은 일을 시키는데 절차가 다른 거예요:

```python
# 옛날 방식 (관공서 양식 — 5줄)
import unittest
class TestSum(unittest.TestCase):
    def test_two_plus_two(self):
        self.assertEqual(sum([2, 2]), 4)

# pytest (포스트잇 — 2줄)
def test_two_plus_two():
    assert sum([2, 2]) == 4
```

추가 마법:
- 🪄 **검사 실패하면 어디가 다른지 자동으로 알려줌** (옛날 도구는 그냥 "틀렸다"만)
- 🪄 **같은 검사를 여러 데이터로 자동 반복** (데코레이터 한 줄)
- 🪄 **DB 같은 무거운 준비물을 여러 검사가 공유**

## 3. 누가 만들었어? — "독일 아저씨가 짜증나서 만들었어요"

만든 사람: **Holger Krekel(홀거 크레켈)**, 독일 개발자.

**언제**: 2003–2004년쯤. 처음 이름은 `py.test`였다가 점이 떨어져서 pytest가 됐어요.

**왜 만들었나**: "**파이썬에 `assert`라는 기본 키워드가 멀쩡히 있는데 왜 `self.assertEqual` 같은 군더더기를 자바처럼 써야 하나?**" 라는 짜증.

핵심 철학 한 줄: **"군더더기 없애자. 그냥 함수랑 assert만 쓰면 끝."**

이 철학이 너무 매력적이라 전 세계 개발자가 모여들었고, 지금은 pytest 관련 도구(플러그인)가 1,500개를 넘어요.

## 4. 나한테 좋은 게 뭐야? — "글 쓰는 양이 70% 줄어요"

**숫자로 비교**:
- 검사 코드 **70% 짧아짐**
- 디버깅 시간 **5분 → 5초**
- 데이터 100개 검사: 메서드 100개 → **함수 1개 + 데코레이터 한 줄**

**실수 방지 사례** (우리 morning-us-index에서):
- 시그널 빨강/초록 표시가 음수일 때 정상인지 검사
- 0으로 나누는 사고가 안 나는지 검사
- 슬랙 통신이 5번 실패하면 멈추는지 검사
- 환경변수 없을 때 에러가 잘 나는지 검사

이 17가지를 pytest가 **0.88초 만에 다 검사**했어요. 우리가 손으로 17번 따라 해보려면 30분 이상 걸려요.

**한 마디로**: 개발자 일상의 **"이거 맞나?"** 라는 불안감을 자동화해줘요. 컴퓨터가 대신 의심해주니, 개발자는 자신 있게 코드를 고칠 수 있어요.

## 5. 안 쓰면 어떻게 돼? — "악순환에 빠져요"

pytest 안 쓰고 옛날 방식 고집하면 이런 악순환이 시작돼요:

```
검사 짜기 귀찮음
    ↓
검사를 적게 짬
    ↓
코드 고칠 때마다 불안함
    ↓
"고치면 뭐가 깨질지 몰라" → 고치는 걸 무서워함
    ↓
오래된 코드는 점점 손도 못 댐
    ↓
어느 날 production에서 폭발 💥
```

**실제 사고 시나리오**:
환율 계산 함수를 살짝 고쳤어요. 옛날 방식 검사 5개 다 통과해서 배포! 그런데 **대만 달러(TWD)가 검사에 없었음**. 한 달 뒤 대만 사용자만 결제 금액이 망가져 있는 걸 발견. 회사 손해 + 신뢰도 ↓.

pytest로 통화 30종을 한 줄에 넣어 검사했으면 PR 단계에서 잡혔을 사고예요.

> 🚨 **마지막 한마디**: 코드 한 줄 짤 때마다 보일러플레이트라는 세금을 내고 살 건가요, 아니면 5분 투자해서 평생 면제받을 건가요? 답은 정해져 있어요.

## 6. 그다음은 뭐야? — "테스트의 품질을 의심하기 시작해요"

pytest를 익히면 자연스럽게 다음 질문이 떠올라요: **"내 검사는 정말 의미 있는 검사인가?"**

| # | 다음 도구 | 한 마디로 | 왜 자연스러운 다음 단계? |
|---|---|---|---|
| 1 | **pytest-mock** | 가짜 친구 만들기 | 외부 API 같은 무거운 의존성을 가짜로 바꿔서 빨리 검사 가능. 우리도 이번에 `@patch("main.requests.post")`로 사용함 |
| 2 | **pytest-cov** | 검사 빈틈 찾기 | "내 코드 100줄 중 몇 줄이 검사를 받고 있나?" 자동으로 측정. 80% 미만이면 위험 |
| 3 | **hypothesis** | AI가 검사 자동 생성 | 사람이 못 떠올리는 경계 케이스를 라이브러리가 수만 개 자동으로 만들어 검사 |
| 4 | **GitHub Actions에 pytest 자동화** | 매일 자동 검사 | PR 올릴 때마다 자동으로 pytest 실행. 사람이 까먹어도 컴퓨터가 안 까먹음 |
| 5 | **mutation testing** | 검사가 진짜 의미 있는지 | 코드를 일부러 한 글자씩 망가뜨려도 검사가 통과하면? 그건 의미 없는 검사라는 진단 |

**4단계 성숙도**:
1. 검사를 **짠다**
2. 검사를 **잘 짠다**
3. 검사가 **자동으로 돌게 한다**
4. 검사 **자체를 의심하고 강화한다**

pytest는 1단계 출발점이자, 4단계 어디서나 중심에 남아 있어요.

## 📚 다른 언어에서 비슷한 것

| 언어 | 비슷한 도구 | 비유 |
|---|---|---|
| Java | JUnit | 가장 원조. pytest가 이걸 보고 "이게 답답해" 하고 만든 것 |
| JavaScript | Jest, Mocha, Vitest | Jest가 가장 인기. 페이스북이 만듦 |
| Ruby | RSpec | "describe, it" 같은 자연어 스타일 |
| Go | testing (표준 라이브러리) | 단순함의 끝판왕 |
| Rust | cargo test | Rust 빌드 도구에 내장 |
| C# | xUnit, NUnit | xUnit은 JUnit의 사촌 |
